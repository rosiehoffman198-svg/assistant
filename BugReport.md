### 2. Время напоминаний сравнивается как строка ✅
**`tools.py:222` + `database.py:88` + `main.py:244`**

```sql
remind_at TEXT NOT NULL                                          -- database.py:88
```
```python

c.execute("... WHERE sent = 0 AND remind_at <= %s", (now,))                      # tools.py:222
```

`remind_at` — TEXT, значение пишется дословно из LLM ([tools.py:198](tools.py#L198)). Формат гарантируется
только текстовым описанием в схеме ([llm.py:161](llm.py#L161)). Валидации/нормализации нет нигде.

**Проверено запуском** при `now = "2026-07-20 09:05"` — все четыре варианта возвращают «не пора»:

| Значение от LLM | `<= now` | Последствие |
|---|---|---|
| `2026-7-20 09:05` (месяц без нуля) | `False` | больше любой даты до 2069 — **не сработает никогда** |
| `2026-07-20 9:05` (час без нуля) | `False` | больше `23:59` — опоздание до суток |
| `2026-07-20T09:05` (ISO `T`) | `False` | `'T'(0x54) > ' '(0x20)` — опоздание на сутки |
| `2026-07-20 09:05:00` (с секундами) | `False` | опоздание на минуту |

Та же проблема в `ORDER BY remind_at` ([tools.py:208](tools.py#L208)) — при смешанных форматах порядок бессмысленный.

### 6. Поиск по заметкам с `'simple'` — нет русской морфологии
**`database.py:80` + `tools.py:174-175`**

```sql

@@ plainto_tsquery('simple', %s)                              -- запрос
```

Конфигурация `'simple'` только приводит к нижнему регистру и режет по пробелам — **без стемминга и без стоп-слов**.

**Сценарий:** в базе 19 сообщений, пользователь быстро шлёт три («привет», «что по задачам», «ок») →

## Проверено и **не** является багом

- **`owner_only` + `functools.wraps`** ([main.py:38-45](main.py#L38-L45)) — работает. `CallableObject` в aiogram 3.13 использует `inspect.getfullargspec`, который **не** следует за `__wrapped__`, поэтому видит реальную сигнатуру обёртки `(message, **kwargs)`, `varkw` выставлен, лишние kwargs принимаются и отбрасываются. Обоснование в комментарии неверно, поведение — корректно.
- **`get_active_projects_summary`** ([tools.py:65-72](tools.py#L65-L72)) — `SELECT p.name, p.goal ... GROUP BY p.id` легален в Postgres по функциональной зависимости от первичного ключа.
- **`send_chat_action(message.chat.id, "typing")`** — позиционные аргументы соответствуют сигнатуре aiogram 3.13.
- **SQL-инъекции** — f-строки в `get_tasks` ([tools.py:123](tools.py#L123)) и `update_project` ([tools.py:54](tools.py#L54)) подставляют только фиксированные имена столбцов; все значения параметризованы.

## Требует проверки на живом API

- **`llm.py:299-308`** — второй запрос передаёт assistant-сообщение с `tool_calls` и `role: "tool"`, но **без** параметра `tools=`. OpenAI-совместимые эндпоинты по-разному строги к этому; Groq исторически отклонял часть таких payload'ов с `400 tool_use_failed`. Не подтверждено вживую. Если сработает — из-за #4 (жёстко прибитая `MODELS[0]`) fallback-пути не будет.

## Что было главным

Цепочка максимального ущерба: **#1** (напоминания мертвы) → **#2** (время как строка) →
**#3** (незащищённый `**func_args`) → **#4** (fallback ломает tool calling).

#3 и #4 особенно неприятны тем, что оба показывали пользователю одинаковое непрозрачное
`❌ Ошибка. Попробуй ещё раз.` — а в случае #4 инструмент **уже закоммитил в БД**, поэтому естественный
повтор запроса молча создавал дубликат. Теперь при отказе второго вызова бот отдаёт результаты
уже выполненных инструментов вместо ошибки, так что повторять запрос не нужно.

**#6** (морфология) и **#7** (несводимые сводки) — те, что тихо обесценивали ядро продукта, поиск и память,
никогда не выбрасывая ошибку.

## Что стоит сделать отдельно

Не баги, а следствия принятых ограничений:

1. **Разовая чистка старых напоминаний.** Строки в неканоническом формате не сработают.
   Посмотреть их: `/reminders` покажет `⚠️`. Если решишь мигрировать базу — достаточно
   `UPDATE reminders SET remind_at = to_char(remind_at::timestamp, 'YYYY-MM-DD HH24:MI')`
   для строк, которые Postgres сумеет привести.
2. **Удалить неиспользуемый индекс** `notes_fts_idx` (`'simple'`), когда будет ясно,
   что база не делится с другим проектом.
3. **Многошаговый tool calling.** Модель делает ровно один раунд вызовов инструментов;
   цепочки вида «найди заметку → создай по ней задачу» за один ход не отработают.
