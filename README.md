# Personal Telegram AI Assistant

Личный AI-ассистент в Telegram: задачи, проекты, заметки, напоминания и долгая память.
Работает только для одного владельца — все чужие сообщения молча игнорируются.

---

## Стек

| Слой | Технология |
|------|-----------|
| Bot | aiogram 3.29+ (long polling), отправка через Rich Messages с откатом на обычные |
| LLM | Groq — `llama-3.3-70b-versatile`, фолбэк `llama-3.1-8b-instant` |
| Голос | Groq Whisper `whisper-large-v3-turbo` (язык: ru) |
| БД | PostgreSQL (`psycopg` 3 + пул соединений), полнотекстовый поиск `tsvector('russian')` + GIN |
| Планировщик | APScheduler (проверка напоминаний раз в минуту) |
| Health check | aiohttp-сервер на `PORT` — нужен для хостинга как Web Service |

## Переменные окружения

```env
TELEGRAM_TOKEN=...        # @BotFather
GROQ_API_KEY=...          # console.groq.com — бесплатно
MY_TELEGRAM_ID=...        # свой numeric ID, узнать у @userinfobot
DATABASE_URL=...          # postgresql://user:pass@host:5432/dbname
TIMEZONE=Europe/Moscow    # опционально, по умолчанию Europe/Moscow
PORT=8080                 # опционально, задаётся хостингом
DB_POOL_MAX=5             # опционально, размер пула соединений
```

Первые четыре обязательны. Без любой из них процесс падает на старте с внятным
сообщением — а не стартует «успешно», молча игнорируя все сообщения.
`.env.example` соответствует этому списку.

## Запуск

```bash
pip install -r requirements.txt

cp .env.example .env
# заполнить по списку выше

python main.py
```

Нужна работающая база PostgreSQL. `init_db()` при старте создаёт только **недостающие**
таблицы, колонки и индексы: ни `DROP`, ни `DELETE`, ни `UPDATE`, ни смены типов колонок.
Повторные запуски безопасны, и база, которую использует что-то ещё, не пострадает.

### Деплой

`Procfile` объявляет процесс как `web: python main.py` — код поднимает HTTP health-check
на `/`, что и требуется хостингу типа **Web Service** (Render и подобные).
Порт берётся из `PORT`.

## Команды

| Команда | Действие |
|---------|----------|
| `/start` | Приветствие и список команд |
| `/tasks` | Активные задачи (до 50, дальше подсказка сузить фильтр) |
| `/done [id]` | Отметить задачу выполненной |
| `/reminders` | Неотправленные напоминания |
| `/search [запрос]` | Полнотекстовый поиск по заметкам с учётом русской морфологии |
| `/profile` | Профиль и закреплённые факты |
| `/setup` | Подсказка, как заполнить профиль |
| `/reset` | Очистить историю диалога и сводки (закреплённые факты остаются) |
| `/forget` | Удалить закреплённые факты |

Всё остальное — свободный текст или голосовое; модель сама решает, вызывать ли инструменты.
Неизвестная команда получает подсказку, а не тишину.

## Инструменты LLM

Модель вызывает их сама через tool calling:

- **Задачи** — `create_task` (priority / importance / energy / deadline / project_id), `get_tasks`, `complete_task`
- **Проекты** — `create_project`, `get_projects`, `update_project`
- **Заметки** — `save_note`, `search_notes`
- **Напоминания** — `create_reminder`, `get_reminders`
- **Память** — `pin_fact`

Системный промпт явно запрещает создавать задачи из приветствий и вопросов —
инструменты только по явному запросу пользователя.

Аргументы от модели не попадают в Python-функции напрямую: [`_prepare_args`](llm.py#L232)
сверяет их с реальной сигнатурой, отбрасывает выдуманные ключи, приводит типы
(`"5"` → `5`, `"false"` → `False`) и возвращает понятное сообщение вместо падения,
если не хватает обязательного поля. Сам вызов обёрнут в [`_run_tool`](llm.py#L261),
который никогда не бросает исключение — ошибка становится текстом для модели.

## Как устроена память

Системный промпт собирается заново на каждый запрос ([`build_system_prompt`](llm.py#L284)) и включает:

1. **Профиль** — имя, цели, стиль общения. При первом текстовом сообщении, если профиль
   пуст, бот пытается извлечь эти поля из текста отдельным быстрым LLM-вызовом.
   Если извлечь нечего, ставится отметка — попытка не повторяется на каждом сообщении.
2. **Активные проекты** — до 5 штук со счётчиком открытых задач.
3. **Закреплённые факты** — последние 20 записей `pinned_facts`.
4. **Сжатый контекст** — последняя сводка прошлых разговоров.

Плюс последние `HISTORY_LIMIT = 6` сообщений диалога.

Каждые `SUMMARY_INTERVAL = 20` новых сообщений в фоне генерируется сводка и сохраняется
в `conversation_summaries` — так контекст не растёт бесконечно. Сводка **кумулятивна**:
предыдущая подаётся модели на вход, поэтому старые факты не теряются при обновлении.
Прогресс отслеживается по `last_message_id`, а не по времени, так что сообщения,
пришедшие во время генерации, не выпадают из памяти. Считается в отдельном потоке
и не задерживает ответ; параллельные запуски отсекаются блокировкой.

## Отправка сообщений

Обычный `sendMessage` ограничен 4096 символами, и длинный `/tasks` в него не помещался.
[`reply`](main.py#L116) сначала пробует **Rich Message** (`sendRichMessage`, Bot API 10.1) —
лимит там на порядок выше, длинный список уходит одним сообщением.

При любом отказе API или слишком большом тексте управление переходит к
[`reply_plain`](main.py#L109): разбивка по 4000 символов по границам строк, без потери текста.
Откат обязателен, потому что точный лимит Rich Messages подтвердить по официальной
документации не удалось, а markdown от LLM бывает битым.

Расшифровка голосового отправляется **только** обычным сообщением и без разметки:
это ненадёжный текст, интерпретировать его как markdown нельзя.

## Надёжность

- **Ничего не блокирует event loop.** Все вызовы Groq и psycopg синхронные, поэтому
  идут через `asyncio.to_thread`. Health-check и планировщик остаются живыми во время
  ответа модели.
- **Соединения не текут.** Пул + контекстный менеджер [`db()`](database.py#L35) гарантируют
  commit/rollback и возврат соединения даже при исключении.
- **Ошибка всегда доходит до пользователя.** Декоратор [`guarded`](main.py#L64) ловит всё,
  что не поймали хендлеры; молчания в ответ на сбой БД больше нет.
- **Ответ сохраняется в историю после доставки**, а не до, — в истории не остаётся
  сообщений, которых пользователь не видел.
- **Инструменты не выполняются дважды.** Если после их выполнения падает финальный вызов
  модели, бот отдаёт результаты уже выполненных инструментов вместо ошибки, чтобы повтор
  запроса не создал дубликаты в базе.

## Схема БД

| Таблица | Что хранит |
|---------|-----------|
| `profile` | key-value профиля (`name`, `goals`, `style`, `extra`; служебные ключи с `_`) |
| `messages` | История диалога |
| `projects` | Проекты: name, goal, status (`active`/`paused`/`done`) |
| `tasks` | Задачи: priority, importance, energy, deadline, project_id |
| `notes` | Заметки с тегами + GIN-индекс `to_tsvector('russian', …)` |
| `reminders` | Напоминания; `remind_at` — TEXT в каноническом `YYYY-MM-DD HH:MM` |
| `pinned_facts` | Факты, которые всегда в промпте |
| `conversation_summaries` | Сводки истории + `last_message_id` |

Время напоминаний нормализуется **перед записью**, а не при сравнении. Фиксированная
ширина с ведущими нулями делает лексикографический порядок равным хронологическому,
поэтому TEXT-колонка сравнивается корректно без миграции в `TIMESTAMP`.

## Поток обработки сообщения

```
Telegram (текст / голос)
   ↓  голос → Groq Whisper (в отдельном потоке) → текст
owner_only — пропускает только MY_TELEGRAM_ID
   ↓
save_message + фоновая генерация сводки
   ↓
build_system_prompt (профиль + проекты + факты + сводка) + последние 6 сообщений
   ↓
Groq chat.completions с TOOL_DEFINITIONS   ← фолбэк на 8b-instant при отказе
   ↓  есть tool_calls → проверка аргументов → выполнение → второй вызов с результатами
   ↓                                          (той же моделью, что сработала)
Rich Message → при отказе обычная отправка с разбивкой
   ↓
save_message("assistant") — после доставки
```

## Ограничения

- **Старые напоминания в кривом формате не сработают.** Строки, созданные до нормализации,
  не переписывались — база может быть общей с другим проектом. В `/reminders` они помечены
  `⚠️`, пересоздай их вручную. Готовый `UPDATE` для чистки — в [BugReport.md](BugReport.md).
- **Rich Messages не проверены вживую.** Ветку отката покрывают тесты, ветку успеха на
  живом Telegram — нет. Требует `aiogram>=3.29.0`; проект был на 3.13.1, так что после
  обновления стоит прогнать основные команды.
- **Один раунд tool calling.** Цепочки вида «найди заметку → создай по ней задачу»
  за один ход не отработают.
- **Голос ограничен русским** — Whisper вызывается с `language="ru"`.
- **Неиспользуемый индекс** `notes_fts_idx` со старой конфигурацией `'simple'` остался
  в базе; удалять его, пока неясно, делится ли база с другим проектом, не стали.

Полный разбор найденных багов и того, как они закрыты, — в [BugReport.md](BugReport.md).
