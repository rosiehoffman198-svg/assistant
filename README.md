# Personal Telegram AI Assistant

Личный AI-ассистент в Telegram: задачи, проекты, заметки, напоминания и долгая память.
Работает только для одного владельца — все чужие сообщения молча игнорируются.

---

## Стек

| Слой | Технология |
|------|-----------|
| Bot | aiogram 3.29+ (long polling), отправка через Rich Messages с откатом на обычные |
| LLM | Groq — `llama-3.3-70b-versatile`, фолбэк `llama-3.1-8b-instant` |
| Голос | Groq Whisper `whisper-large-v3-turbo` (язык: ru) |
| БД | PostgreSQL (`psycopg` 3 + пул соединений), полнотекстовый поиск `tsvector('russian')` + GIN |
| Планировщик | APScheduler (проверка напоминаний раз в минуту) |
| Health check | aiohttp-сервер на `PORT` — нужен для хостинга как Web Service |

---

## Файлы

| Файл | Назначение |
|------|-----------|
| `main.py` | Точка входа: хендлеры команд, голос/текст, планировщик, health-сервер |
| `llm.py` | Системный промпт, определения инструментов, вызов Groq, tool calling, сжатие истории |
| `tools.py` | Вся работа с БД: проекты, задачи, заметки, напоминания, факты, профиль, история |
| `database.py` | Подключение к Postgres и создание/миграция схемы |
| `config.py` | Чтение переменных окружения и константы моделей |

## Переменные окружения

```env
TELEGRAM_TOKEN=...        # @BotFather
GROQ_API_KEY=...          # console.groq.com — бесплатно
MY_TELEGRAM_ID=...        # свой numeric ID, узнать у @userinfobot
DATABASE_URL=...          # postgresql://user:pass@host:5432/dbname
TIMEZONE=Europe/Moscow    # опционально, по умолчанию Europe/Moscow
PORT=8080                 # опционально, задаётся хостингом
DB_POOL_MAX=5             # опционально, размер пула соединений
```

Первые четыре обязательны — без любой из них процесс падает на старте
с внятным сообщением, а не молча игнорирует все сообщения.
`.env.example` соответствует этому списку.

---

## Запуск

```bash
pip install -r requirements.txt

cp .env.example .env
# заполнить по списку выше, обязательно добавить DATABASE_URL

python main.py
```

Нужна работающая база Postgres — схема создаётся автоматически при старте
(`init_db()` идемпотентен, повторные запуски безопасны).

### Деплой

`Procfile` объявляет процесс как `web: python main.py` — код поднимает HTTP health-check
на `/`, что и требуется хостингу типа **Web Service** (Render и подобные).
Порт берётся из `PORT`.

`init_db()` создаёт только недостающие таблицы, колонки и индексы. Он не меняет типы
колонок и не трогает существующие строки, поэтому безопасен для базы,
которую использует что-то ещё.

## Команды

| Команда | Действие |
|---------|----------|
| `/start` | Приветствие и список команд |
| `/tasks` | Активные задачи |
| `/done [id]` | Отметить задачу выполненной |
| `/reminders` | Неотправленные напоминания |
| `/search [запрос]` | Полнотекстовый поиск по заметкам |
| `/profile` | Профиль и закреплённые факты |
| `/setup` | Подсказка, как заполнить профиль |
| `/reset` | Очистить историю диалога и сводки |
| `/forget` | Удалить закреплённые факты |

Всё остальное — свободный текст или голосовое; модель сама решает, вызывать ли инструменты.

---

## Инструменты LLM

Модель вызывает их сама через tool calling:

- **Задачи** — `create_task` (priority / importance / energy / deadline / project_id), `get_tasks`, `complete_task`
- **Проекты** — `create_project`, `get_projects`, `update_project`
- **Заметки** — `save_note`, `search_notes`
- **Напоминания** — `create_reminder`, `get_reminders`
- **Память** — `pin_fact`

Системный промпт явно запрещает создавать задачи из приветствий и вопросов —
инструменты только по явному запросу пользователя.

---

## Как устроена память

Системный промпт собирается заново на каждый запрос ([`build_system_prompt`](llm.py#L206)) и включает:

1. **Профиль** — имя, цели, стиль общения. При первом текстовом сообщении, если профиль
   пуст, бот пытается извлечь эти поля из текста отдельным LLM-вызовом.
2. **Активные проекты** — до 5 штук со счётчиком открытых задач.
3. **Закреплённые факты** — последние 20 записей `pinned_facts`.
4. **Сжатый контекст** — последняя сводка прошлых разговоров.

Плюс к промпту добавляются последние `HISTORY_LIMIT = 6` сообщений диалога.

Каждые `SUMMARY_INTERVAL = 20` новых сообщений в фоне генерируется сводка и сохраняется
в `conversation_summaries` — так контекст не растёт бесконечно. Сводка **кумулятивна**:
предыдущая подаётся модели на вход, поэтому старые факты не теряются при обновлении.
Прогресс отслеживается по `last_message_id`, а не по времени, так что сообщения,
пришедшие во время генерации, не выпадают из памяти. Считается в отдельном потоке
и не задерживает ответ; параллельные запуски отсекаются блокировкой.

## Схема БД

| Таблица | Что хранит |
|---------|-----------|
| `profile` | key-value профиля (name, goals, style, extra) |
| `messages` | История диалога |
| `projects` | Проекты: name, goal, status (`active`/`paused`/`done`) |
| `tasks` | Задачи: priority, importance, energy, deadline, project_id |
| `notes` | Заметки с тегами + GIN-индекс для поиска |
| `reminders` | Напоминания с флагом `sent` |
| `pinned_facts` | Факты, которые всегда в промпте |
| `conversation_summaries` | Сжатые сводки истории |

## Поток обработки сообщения

```
Telegram (текст / голос)
   ↓  голос → Groq Whisper → текст
owner_only — пропускает только MY_TELEGRAM_ID
   ↓
save_message + фоновая генерация сводки
   ↓
build_system_prompt (профиль + проекты + факты + сводка) + последние 6 сообщений
   ↓
Groq chat.completions с TOOL_DEFINITIONS
   ↓  если есть tool_calls → выполнить → второй вызов с результатами
Ответ → Telegram
```

При ошибке основной модели идёт автоматический фолбэк на `llama-3.1-8b-instant`.

## Ограничения

- **Старые напоминания в кривом формате не сработают.** Время хранится в TEXT-колонке;
  новые записи нормализуются в `YYYY-MM-DD HH:MM` перед сохранением, но строки, созданные
  до этого, не переписывались (база может быть общей с другим проектом). В `/reminders`
  они помечены `⚠️` — пересоздай их вручную.
- **Один раунд tool calling.** Цепочки вида «найди заметку → создай по ней задачу»
  за один ход не отработают.
- **Голос ограничен русским** — Whisper вызывается с `language="ru"`.
- **Rich Messages не проверены вживую.** Ответы уходят через `sendRichMessage`
  (Bot API 10.1), при отказе — обычной отправкой с разбивкой по 4000 символов.
  Ветку отката покрывают тесты, ветку успеха на живом Telegram — нет.
  Требует `aiogram>=3.29.0`; проект был на 3.13.1, так что после обновления
  стоит прогнать основные команды.

Полный разбор найденных багов и того, как они закрыты, — в [BugReport.md](BugReport.md).
